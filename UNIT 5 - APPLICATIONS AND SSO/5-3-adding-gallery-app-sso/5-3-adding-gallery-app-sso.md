## Why This Matters

*SC-300: Implement access management for applications — Configure single sign-on for applications*

This is where theory meets practice. Modules 5.1 and 5.2 taught *what* the objects are. This module teaches *how to use them*. Single Sign-On means the user signs in once to Entra ID, and that sign-in is trusted by the SaaS app. No second password. No separate credentials to manage. This is how modern organizations work.

---

## 🔑 What Is Single Sign-On, Really?

Users think SSO means "sign in with Microsoft credentials." Technically, it's more precise:

**SSO is a federation agreement between two systems:**
- Entra ID: "If I verify this is Alex Lee at Contoso, will you trust me?"
- Slack: "Yes. If you say Alex is verified, I'll let her in."

The mechanism: Entra ID issues a cryptographically signed assertion (SAML, OIDC, or WS-Fed). Slack reads the assertion, verifies the signature, and grants access.

**What the user sees:** Click "Sign in with Contoso," and immediately they're in. No password needed for Slack.

**What the user doesn't see:** A multi-step exchange of tokens and assertions in the background.

---

## 📱 The Two SSO Protocols: SAML vs OIDC

| | **SAML** | **OIDC** |
|---|---|---|
| **What it is** | Security Assertion Markup Language — XML-based federation | OpenID Connect — HTTP-based, modern |
| **When it's used** | Enterprise SaaS (Salesforce, ServiceNow, Workday) | Modern apps, APIs, custom apps |
| **Setup complexity** | More steps (download certificates, configure endpoints) | Simpler setup, JSON-based |
| **Which one Slack uses** | SAML (as of 2025) | Some newer apps use OIDC |

For this module, you'll set up SAML SSO for Slack — the most common scenario in enterprise environments.

---

## ⚙️ The Four Things You Must Configure for SAML SSO

Every SAML SSO setup requires four pieces:

| Element | What it is | Where you get it |
|---|---|---|
| **Entity ID** | Identifies who Slack is (URL format) | From Slack's settings |
| **ACS URL** | Where Slack receives assertions from Entra ID | From Slack's settings |
| **Signing certificate** | Entra ID's certificate to sign assertions | Generated by Entra ID or upload your own |
| **Attribute claims** | Maps Entra ID user properties to Slack fields | You configure: `email` → `Email`, `name` → `Full Name` |

If any one of these is missing or wrong, SSO fails silently — the user sees "authentication failed" with no helpful error message.

## Lab 5.3.1 — Add Slack from the Gallery and Start SSO Setup

**Tools:** Entra admin center, Slack admin account
**Prerequisite:** Module 5.1 complete (you may have already added Slack in Lab 5.1.1)

**Context:** Contoso wants Slack SSO enabled so employees don't manage a separate password. Alex Lee should be able to sign in to Slack by clicking "Sign in with Contoso" and using their Microsoft credentials.

---

1. Navigate to **Applications** → **Enterprise applications** → **+ New application**.

2. Search for **Slack**. Click on **Slack** in the results.

3. On the Slack app preview, click **Create**.

4. Entra ID creates the Enterprise Application. You should see: "Slack has been successfully created." A notification appears with a link to configure single sign-on.

5. Click the **Configure single sign-on** link, or navigate to **Applications** → **Enterprise applications** → **Slack** → left menu **Single sign-on**.

6. A panel opens asking which SSO method to use. Select **SAML** (most common for Slack).

7. Click **Continue**. You're now on the **Set up Single Sign-On with SAML** page.

**You'll know this worked when:** You can see the SAML setup page with sections for:
- Basic SAML Configuration
- User Attributes & Claims
- SAML Signing Certificate
- Set up Slack

## Lab 5.3.2 — Configure Basic SAML Details (Entra ID Side)

**Tools:** Entra admin center, Slack admin settings (you'll need Slack admin credentials)
**Prerequisite:** Lab 5.3.1 complete

This lab requires **Slack admin credentials**. If you don't have a Slack account, create a free Slack workspace at slack.com. You'll need admin access to configure SSO settings.

**Step 1: Get the Entity ID and ACS URL from Slack**

1. Sign in to Slack as an admin: `https://<workspace>.slack.com/admin`

2. Navigate to **Settings** → **Authentication** → **Single Sign-On** (or similar path; Slack UI changes).

3. Look for **Assertion Consumer Service (ACS) URL** and **Entity ID** (or "Service Provider Issuer"). These are values Slack provides.

4. Note these values. You'll paste them into Entra ID.

**Step 2: Configure Entra ID's Basic SAML Configuration**

5. In the Entra admin center, on the **Set up Single Sign-On with SAML** page, find **Basic SAML Configuration** and click **Edit** (pencil icon).

6. Fill in:
   - **Identifier (Entity ID):** `https://slack.com` (usually Slack provides a specific format; use what Slack provides)
   - **Reply URL (Assertion Consumer Service URL):** Paste the ACS URL from Slack's settings
   - **Sign on URL:** `https://<workspace>.slack.com` (your Slack workspace URL)
   - **Relay State:** Leave empty (advanced feature)

7. Click **Save**.

**Step 3: Verify Signing Certificate**

8. Scroll down to **SAML Signing Certificate** section. A certificate is auto-generated.

9. Note the **Certificate Thumbprint** (a unique identifier for this certificate). You'll need this in Slack.

10. Optional: Download the certificate and Federation Metadata XML — you might need these for Slack configuration.

## Lab 5.3.3 — Map User Attributes and Assign Users

**Tools:** Entra admin center
**Prerequisite:** Lab 5.3.2 complete

**Step 1: Configure Attribute Claims**

1. On the **Set up Single Sign-On with SAML** page, find **User Attributes & Claims** and click **Edit**.

2. You'll see a list of claims Entra ID sends to Slack. The default includes:
   - `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name` → `user.displayname`
   - `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress` → `user.mail`
   - `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname` → `user.givenname`
   - `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname` → `user.surname`

3. These defaults are usually correct. Do not change them unless Slack's documentation specifies different claim names.

4. Click **Save**.

**Step 2: Assign Users to the App**

5. Navigate to **Slack** → left menu **Users and groups** → **+ Add user/group**.

6. Select **Alex Lee** (or another test user). Click **Assign**.

7. Repeat for any other users who should have Slack access. For now, assign Alex and Morgan.

8. You should see both users listed with assignment status.

> 💡 **Tip:** In a real organization, you might assign a group (e.g., "All Finance users") instead of individuals. Groups scale better.

## Lab 5.3.4 — Test SSO from the User Perspective

**Tools:** InPrivate browser, Slack workspace
**Prerequisite:** Lab 5.3.3 complete, and SSO configured in your Slack admin settings

**Context:** You've configured Entra ID to send SSO assertions to Slack. Now test that a user can actually sign in.

1. Open an **InPrivate** browser window (important: separate from your admin session).

2. Go to `https://<workspace>.slack.com` (your Slack workspace URL).

3. You should see a Slack sign-in page. Look for a link like "Sign in with Contoso" or "Sign in with your organization" or a button linking to your identity provider.

4. Click the SSO link. You're redirected to the Entra ID sign-in page.

5. Sign in as **alex.lee@yourdomain.onmicrosoft.com** with password `Contoso@1234!` (or the password Alex updated in earlier modules).

6. If MFA is required, complete it.

7. You're redirected back to Slack. If everything is configured correctly, you're now signed in as Alex. You'll see the Slack workspace.

8. If you see an error like "SAML assertion validation failed" or "User not found":
   - Check that Alex's email address matches what Slack expects (usually `user.mail` attribute)
   - Verify the SAML signing certificate is correct
   - Check the Slack and Entra ID logs for the specific error

**You'll know this worked when:** You can sign in to Slack using your Contoso Entra ID credentials without being prompted for a Slack password.

## Lab 5.3.5 — Verify the SSO Configuration in Entra ID

**Tools:** Entra admin center
**Prerequisite:** Lab 5.3.4 complete (successful SSO test)

After testing, verify the configuration is working:

1. Navigate to **Applications** → **Enterprise applications** → **Slack**.

2. On the **Single sign-on** page, scroll to **Set up Slack** section.

3. Click **Test** (if available). A new window opens with instructions to test the SSO flow in a controlled way.

4. Alternatively, navigate to **Users and groups** on Slack's page. Confirm Alex Lee is assigned.

5. Navigate to **Properties**. Confirm the app is visible and accessible:
   - **User can sign in**: Yes
   - **Assignment required**: You choose (if Yes, only assigned users can access; if No, anyone in the tenant can access)

For Slack, you typically set **Assignment required** to **Yes** so only intended users have access.

## Common Mistakes

**Not downloading Slack's SAML metadata before configuring Entra ID.**
Slack provides Entity ID and ACS URL in admin settings. If you don't note these *first*, you end up guessing what values to put in Entra ID, and SSO fails. Always get Slack's configuration values before you start configuring Entra ID.

**Changing default attribute claims when Slack doesn't require it.**
The default claims (name, email, givenname) work for most SaaS apps. Customizing them without understanding what Slack expects creates hard-to-debug failures. Slack's documentation specifies which claims it requires. Only deviate from defaults if the documentation says to.

**Forgetting to assign users after configuring SSO.**
You configure SAML SSO but don't assign any users. When Alex tries to sign in, she gets "not found" — because Entra ID doesn't have her in the app's scope. Assign users and groups in **Users and groups**.

**Testing with your admin account instead of a test user.**
You're signed in as Global Administrator in the admin portal. When you test SSO, you might see different behavior than a regular user. Always test in a separate browser (InPrivate) signed in as a *regular user* (like Alex Lee) to see the actual user experience.
